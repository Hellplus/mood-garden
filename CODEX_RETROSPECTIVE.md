# Mood Garden 与 Codex 学习复盘

这份文档记录我通过 Mood Garden 情绪花园项目学习使用 Codex 的全过程。它不是商业宣传，更像一次个人学习复盘：我从一个空项目开始，借助 Codex 一步步完成需求分析、代码实现、功能验收、视觉优化、文档整理和发布准备。

## 项目起点：从没有现成项目开始

Mood Garden 一开始没有现成代码，也没有项目结构。最初的想法只是做一个小型前端项目：用户可以选择今天的情绪，写一句心情，然后生成一张情绪花卡片。

项目从最基础的三个文件开始：

- `index.html`
- `style.css`
- `script.js`

一开始没有框架、没有构建工具、没有后端，也没有复杂设计。这个起点让项目更适合作为原生前端练习，也更方便观察 Codex 如何从零协助搭建项目。

## 项目目标：用 Codex 从 0 做一个有创意的小作品

这个项目的目标不是做一个商业级产品，而是练习如何和 Codex 协作完成一个完整的小作品。

我希望它具备几个特点：

- 主题有一点创意，而不是普通表单练习
- 功能可以逐步扩展
- 代码适合新手阅读
- 保持 HTML、CSS、JavaScript 三文件主体结构
- 能发布到 GitHub Pages
- 最终可以放进个人作品集展示

Mood Garden 的主题是“把心情种成一朵花”。这个方向让项目既能练习数据和交互，也能练习视觉风格、动画和文案。

## 主要迭代阶段

### V1.0 基础情绪记录

这一阶段完成最小可运行版本：

- 显示项目标题
- 选择情绪
- 输入一句心情
- 点击按钮生成花卡片
- 使用 `localStorage` 保存历史记录
- 刷新后恢复花卡片

这个阶段让我理解了从 0 创建一个前端项目时，先做最小闭环比一开始追求完整功能更重要。

### V1.1 浏览管理

这一阶段增加了浏览历史记录的能力：

- 按情绪筛选
- 按关键词搜索
- 最新优先 / 最早优先排序
- 一键重置筛选
- 显示当前筛选结果数量

这一阶段的重点是区分“原始数据”和“当前显示数据”。筛选和排序只影响页面显示，不应该改变 `localStorage` 里的原始记录。

### V1.2 数据备份

这一阶段增加数据备份能力：

- 导出 JSON 备份
- 导入 JSON 备份
- 导入前预览
- 覆盖导入 / 合并导入
- 错误格式提示
- 兼容旧数据

这个阶段让我开始关注数据结构、格式校验、错误提示和旧数据兼容。即使是小项目，也需要考虑用户数据不要轻易丢失。

### V1.3 视觉体验

这一阶段主要优化页面体验：

- 浅色、深色、治愈粉三种主题
- 主题偏好保存
- Toast 轻提示
- 空状态和无结果状态
- 页面布局重设计
- 花卡片视觉升级
- 移动端适配
- 页面、卡片、Toast、Hero 的轻量动画

这个阶段让我看到，前端项目不只是功能能用，还需要考虑层级、节奏、反馈和移动端体验。

### V1.4 数据分析、动态图标和 Hero 动态文案

这一阶段增加了更多数据和表现层能力：

- 最近 7 天记录统计
- 情绪分布
- 最常出现的情绪
- 连续记录天数
- 本周情绪回顾
- 情绪图标池
- 图标专属动画
- Hero 动态图标
- Hero 动态文案

这个阶段的重点是：分析只基于已有记录做轻量总结，不做心理诊断。动态图标和 Hero 文案也只是展示层，不应该改变核心数据结构。

### V1.5 代码整理

随着功能越来越多，代码也变复杂。V1.5 的目标不是新增功能，而是整理项目：

- 集中配置常量
- 统一 `localStorage` 读写
- 整理渲染刷新流程
- 保留旧数据兼容逻辑
- 补充基础可访问性
- 增强错误处理

这个阶段让我意识到，项目越到后期，整理代码比继续堆功能更重要。

### V1.6 发布质量

这一阶段把项目整理成更适合公开展示的 GitHub Pages 作品：

- 优化浏览器标题
- 增加 `meta description`
- 增加项目版本号
- 增加页面使用说明和 footer
- 优化 README
- 新增 CHANGELOG
- 检查本地路径、外部依赖和发布风险

这个阶段让我学习到，一个项目“能运行”和“适合展示”之间还有一段距离。

### V1.7 最终展示

这一阶段主要完善作品展示资料：

- 增加项目截图
- 优化 README 为 GitHub 首页展示版
- 新增 PROJECT_BRIEF
- 新增 RELEASE_CHECKLIST
- 生成最终作品介绍文案
- 检查文档是否真实、不夸大

这个阶段更像最后的收尾，让项目不只是代码文件，也有完整的说明、版本记录和发布检查清单。

## 我是如何控制 Codex 的

### 先分析，不直接改代码

在大多数功能开始前，我会先要求 Codex 只做分析，不修改文件。这样可以先理解当前项目结构、数据流和风险点。

这种方式能避免 Codex 过早动手，减少不必要的大改。

### 先出方案，再批准执行

对于新增功能，我通常会先让 Codex 给出修改方案，包括：

- 需要修改哪些文件
- HTML 需要增加什么结构
- CSS 需要增加什么样式
- JavaScript 需要新增哪些函数
- 如何测试

确认方案后，我再让 Codex 开始写代码。

### 限制修改范围

每次修改时，我都会明确限制范围，例如：

- 不重构整个项目
- 不引入第三方库
- 不改变 `localStorage` 主结构
- 只修改 CSS
- 只修改 README
- 只新增一个文件

这些限制让 Codex 的修改更可控，也更适合新手项目维护。

### 每次修改后做本地测试

功能完成后，我会要求 Codex告诉我如何测试，或者让它按验收清单检查。

测试重点通常包括：

- 新功能是否正常
- 旧功能是否被影响
- 刷新后数据是否恢复
- 导入导出是否正常
- 空状态和错误提示是否友好

### 让 Codex 做自检

每个重要阶段完成后，我会让 Codex 做一次“只分析，不修改文件”的验收。

这样可以发现：

- 明确 bug
- 小范围修复建议
- 是否有重复逻辑
- 是否影响旧功能
- 是否有发布风险

### 用 GitHub 保存版本

项目适合通过 GitHub 保存每个阶段的版本。这样可以在功能迭代过程中保留历史记录，也方便发布到 GitHub Pages。

即使项目很小，用版本管理也能降低试错成本。

## 我学到的前端知识

通过这个项目，我练习了：

- HTML 页面结构和语义分区
- CSS 布局、响应式设计和主题变量
- CSS 动画和 `prefers-reduced-motion`
- JavaScript DOM 查询、事件监听和动态渲染
- 使用数组和对象管理数据
- 使用 `localStorage` 保存和读取本地数据
- 使用 `JSON.stringify` 和 `JSON.parse`
- 生成 txt 和 JSON 下载文件
- 读取用户上传的 JSON 文件
- 表单校验和错误提示
- 基础可访问性，例如 label、aria-label、aria-live
- GitHub Pages 静态页面发布

## 我学到的项目管理方法

这个项目让我练习了小步迭代：

- 先做最小可运行版本
- 每次只加一组相关功能
- 功能完成后做验收
- 有问题时只做小范围修复
- 定期整理代码和文档
- 最后再做发布质量和展示材料

我也学到，项目越往后，越需要控制范围。否则一个小项目很容易因为不断加功能而变得难以维护。

## 我学到的 Codex 使用方法

我发现和 Codex 协作时，清晰的任务描述很重要。比较有效的方式包括：

- 明确“先分析，不要改代码”
- 明确“方案可以，开始修改”
- 明确“只修改某个文件”
- 明确“不新增功能、不重构”
- 明确“修改后告诉我改了什么”
- 明确“做一次验收，只分析”

Codex 比较适合做：

- 需求拆解
- 小范围代码实现
- 重复性文档整理
- 功能验收清单
- README 和项目说明
- 代码结构审查

但我仍然需要判断方向、控制范围、确认结果，并决定哪些建议值得做。

## 以后做新项目时可以复用的流程

以后做新项目时，可以继续复用这套流程：

1. 先描述项目目标和限制。
2. 让 Codex 给出技术方案和文件结构。
3. 先实现最小可运行版本。
4. 每次只迭代一组功能。
5. 每次修改前先分析，确认后再执行。
6. 每次修改后做验收。
7. 明确区分 bug 修复、功能新增、视觉优化和代码整理。
8. 定期更新 README。
9. 发布前新增 CHANGELOG、项目说明和检查清单。
10. 最后做一次公开展示审查。

## 这个项目目前的局限

Mood Garden 目前仍然是一个轻量前端项目，有一些明显局限：

- 数据只保存在当前浏览器中，没有账号系统和云同步
- 没有后端数据库，换设备后无法自动同步
- 数据分析比较基础，不适合做严肃心理判断
- 没有完整自动化测试
- 功能较多，虽然已整理，但仍然集中在一个 `script.js` 文件中
- 截图和文档需要随着项目更新手动维护

这些局限是刻意接受的，因为项目目标是原生前端练习和作品展示，而不是完整商业应用。

## 后续可以继续升级的方向

后续可以考虑：

- 增加日历视图
- 增加按日期范围筛选
- 增加标签、天气或地点字段
- 增加更完整的键盘可访问性
- 增加自动化测试
- 增加更清晰的导入结果报告
- 压缩 README 截图体积
- 尝试用 React 或 Vue 重写一版，用来练习框架开发

如果继续升级，仍然需要控制范围，避免把一个轻量作品变成难以维护的大项目。
