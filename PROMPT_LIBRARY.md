# Codex 提示词模板库

这份模板库整理自 Mood Garden 情绪花园项目的实践经验。以后做新项目时，可以按场景复制使用，再把项目名称、功能范围和文件名替换成自己的内容。

## 1. 项目规划模板

### 使用场景

从 0 开始一个新项目，还没有文件结构，也不确定用什么技术实现。

### 提示词

```text
我想从零创建一个项目，项目名叫【项目名称】。

项目目标：
【用几句话描述项目想做什么】

请你先不要写代码。
请先告诉我：
1. 推荐用什么技术实现；
2. 需要创建哪些文件；
3. 每个文件负责什么；
4. 第一个最小可运行版本应该包含哪些功能；
5. 后续可以如何迭代。

请用中文回答。
```

### 适合什么时候用

适合项目刚开始时使用。这个模板可以帮助先理清方向，避免一开始就写出混乱的代码。

## 2. 只分析不修改模板

### 使用场景

想让 Codex 检查当前项目、分析问题或评估可行性，但暂时不希望它修改文件。

### 提示词

```text
请先分析当前项目，不要修改任何文件。

请重点检查：
1. 【检查点 1】；
2. 【检查点 2】；
3. 【检查点 3】；
4. 是否存在明显问题；
5. 如果要修改，建议怎么做。

请只做分析，不要写代码。
```

### 适合什么时候用

适合新增功能前、重构前、发布前、发现问题但还不确定原因时使用。

## 3. 功能包开发模板

### 使用场景

一次开发一组相关功能，例如筛选搜索排序、导入导出、主题系统、数据分析等。

### 提示词

```text
我想实现【版本号 / 功能包名称】。

这次功能包包括：
1. 【功能 1】；
2. 【功能 2】；
3. 【功能 3】。

重要要求：
1. 不影响已有功能；
2. 不引入第三方库；
3. 不重构整个项目；
4. 不改变已有数据结构；
5. 代码保持适合新手阅读。

请先说明完整修改方案，不要直接修改代码。

请重点说明：
- 需要修改哪些文件；
- HTML 准备新增什么；
- CSS 准备新增什么；
- JavaScript 准备新增哪些变量和函数；
- 如何兼容旧数据；
- 如何完整测试。
```

### 适合什么时候用

适合功能有一定规模，但又不想一次性大重构时使用。先让 Codex 出方案，确认后再执行。

## 4. 小范围修复模板

### 使用场景

验收后发现明确问题，只想修 bug，不想让 Codex 顺手加新功能。

### 提示词

```text
请只修复刚才验收中发现的明确问题。

要求：
1. 不新增功能；
2. 不重构项目；
3. 不影响已有功能；
4. 只修改和问题相关的文件；
5. 修改完成后告诉我改了哪些文件，以及我应该如何验证。
```

### 适合什么时候用

适合功能验收后的小问题修复。这个模板能防止修 bug 变成大改。

## 5. 代码审查模板

### 使用场景

想检查代码结构、重复逻辑、命名、错误处理、可维护性。

### 提示词

```text
请对当前项目做一次代码审查。

要求：
1. 只分析，不要修改文件；
2. 重点检查：
   - 是否存在重复函数；
   - 是否存在重复逻辑；
   - 数据读写是否清晰；
   - 渲染流程是否清楚；
   - 命名是否容易理解；
   - 是否有明显 bug；
   - 是否有可以小幅优化但不需要大重构的地方；
3. 如有问题，只列出问题和修复建议。
```

### 适合什么时候用

适合功能逐渐变多、准备整理代码或发布前使用。

## 6. UI 审查模板

### 使用场景

想检查页面设计、视觉层级、按钮层级、移动端体验和整体观感。

### 提示词

```text
请对当前项目做一次 UI / 页面设计审查。

要求：
1. 只分析，不要修改文件；
2. 从真实用户打开页面的角度检查；
3. 重点检查：
   - 页面整体是否完整；
   - 首屏是否清楚；
   - 功能区是否堆叠；
   - 按钮层级是否混乱；
   - 卡片是否有识别度；
   - 空状态是否友好；
   - 移动端是否拥挤；
   - 哪些可以小范围优化；
   - 哪些不应该动。

请输出问题和建议。
```

### 适合什么时候用

适合功能基本完成后，准备提升视觉完成度时使用。

## 7. 发布质量审查模板

### 使用场景

准备把项目放到 GitHub Pages、作品集或简历中展示。

### 提示词

```text
请对当前项目做一次发布质量审查。

要求：
1. 只分析，不要修改文件；
2. 重点检查：
   - 浏览器 title 是否清楚；
   - 是否有 meta description；
   - README 是否适合 GitHub 展示；
   - 是否需要 CHANGELOG；
   - 是否说明数据保存方式；
   - 是否有本地路径或无效链接；
   - 是否有隐私信息；
   - 移动端是否适合作品展示；
   - 是否有发布风险。

请输出：
- 已经做得好的地方；
- 最需要补齐的发布项；
- 推荐现在做的修改；
- 不建议现在做的修改；
- 上线测试清单。
```

### 适合什么时候用

适合功能完成后、正式发布前使用。

## 8. README 更新模板

### 使用场景

功能完成后，需要更新 README，让项目首页说明和代码保持一致。

### 提示词

```text
请只修改 README.md，补充最新功能说明。

要求：
1. 不修改其他代码文件；
2. 保留现有线上访问地址；
3. 内容真实、清楚，不要夸大；
4. 适合 GitHub 项目展示；
5. 需要补充：
   - 【新增功能 1】；
   - 【新增功能 2】；
   - 使用方式；
   - 学习收获；
   - 注意事项。

修改完成后告诉我具体改了哪些部分。
```

### 适合什么时候用

适合每个功能包完成后，或者发布前整理 README 时使用。

## 9. CHANGELOG 更新模板

### 使用场景

项目进入阶段性版本，需要记录版本变化。

### 提示词

```text
请只修改 CHANGELOG.md，补充【版本号】版本说明。

要求：
1. 不修改其他文件；
2. 内容简洁、真实；
3. 记录主要变化，不写太细；
4. 包含：
   - 新增了什么；
   - 优化了什么；
   - 修复或整理了什么；
   - 是否影响数据结构。

修改完成后告诉我改了哪些部分。
```

### 适合什么时候用

适合完成一个版本后使用。不要把每个小修复都写得太长。

## 10. GitHub Pages 发布前检查模板

### 使用场景

准备上线或更新 GitHub Pages 前，做最后一次发布检查。

### 提示词

```text
请对当前项目做一次 GitHub Pages 发布前检查。

要求：
1. 只分析，不要修改文件；
2. 检查：
   - 是否使用相对路径引用 CSS、JS、图片；
   - 是否存在本地绝对路径；
   - 是否存在无效外部链接；
   - README 在线访问地址是否正确；
   - 页面 title 和 meta description 是否完整；
   - 是否有隐私信息；
   - 是否有不适合公开的内容；
   - 是否需要补充发布说明。

如有问题，只列出问题和修复建议。
```

### 适合什么时候用

适合发布前、部署后、更新截图或文档后使用。

## 11. 完整回归测试模板

### 使用场景

完成一个功能包后，需要确认新功能和旧功能都没有坏。

### 提示词

```text
请对当前项目做一次完整回归测试审查。

要求：
1. 只分析，不要修改文件；
2. 检查新增功能是否正常；
3. 检查已有功能是否仍然正常；
4. 检查刷新恢复、导入导出、空状态、错误提示；
5. 检查移动端和主题切换；
6. 检查是否有明显重复代码或小问题；
7. 如有问题，只列出问题和修复建议。

重点功能：
- 【功能 1】；
- 【功能 2】；
- 【功能 3】。
```

### 适合什么时候用

适合每个版本完成后使用，尤其是修改了数据、渲染或导入导出逻辑之后。

## 12. 不允许重构模板

### 使用场景

只想做一个小功能或修复，不希望 Codex 大范围整理代码。

### 提示词

```text
请严格控制修改范围。

要求：
1. 不重构整个项目；
2. 不新增无关功能；
3. 不引入第三方库；
4. 不改变现有数据结构；
5. 不删除已有功能；
6. 只修改完成当前需求所必需的代码；
7. 代码保持当前项目风格。
```

### 适合什么时候用

适合项目已经比较稳定时使用。它能减少“顺手大改”的风险。

## 13. 兼容旧数据模板

### 使用场景

新增字段、调整数据结构或导入旧备份时，需要保护旧记录。

### 提示词

```text
这次修改需要兼容旧数据。

要求：
1. 旧记录如果缺少新字段，页面不能报错；
2. 可以提供默认值或临时显示 fallback；
3. 不要强行破坏旧数据；
4. 不要改变 localStorage 主结构；
5. 导入旧备份时也要兼容；
6. 请说明旧数据兼容逻辑在哪里实现。
```

### 适合什么时候用

适合新增 `id`、`createdAt`、`flowerQuote`、`moodIcon` 等字段时使用。

## 14. localStorage 数据保护模板

### 使用场景

项目使用浏览器本地存储，需要避免误删、误覆盖或数据格式损坏。

### 提示词

```text
请特别注意 localStorage 数据保护。

要求：
1. 不改变现有 localStorage key；
2. 不改变主数据结构；
3. 读取失败时不要让页面崩溃；
4. 保存失败时给用户提示；
5. 清空或覆盖导入前必须二次确认；
6. 筛选、搜索、排序只能影响显示，不改变原始数据；
7. 导入导出要保留已有字段；
8. 修改完成后说明 localStorage 读写是否有变化。
```

### 适合什么时候用

适合任何涉及本地保存、导入导出、清空、编辑、删除、排序的功能。

## 使用建议

- 大功能先用“功能包开发模板”。
- 不确定问题先用“只分析不修改模板”。
- 修改后用“完整回归测试模板”。
- 发现问题后用“小范围修复模板”。
- 发布前用“发布质量审查模板”和“GitHub Pages 发布前检查模板”。
- 涉及数据时一定加上“兼容旧数据模板”和“localStorage 数据保护模板”。
