# Codex 项目工作流手册

这份文档总结我以后使用 Codex 做项目时可以复用的工作流。核心原则是：先分析，再方案，再执行；每次控制范围；每个阶段都测试和保存。

## 一、启动新项目时怎么问 Codex

启动新项目时，不要一开始就让 Codex 直接写代码。先让它帮我分析需求、拆解功能、规划技术方案和文件结构。

可以这样问：

```text
我想从零创建一个项目，项目目标是……

请先不要写代码。
请先告诉我：
1. 推荐用什么技术实现；
2. 需要创建哪些文件；
3. 每个文件负责什么；
4. 第一个最小可运行版本应该包含哪些功能。
```

推荐流程：

1. 先描述项目目标和用户场景。
2. 让 Codex 分析需求。
3. 让 Codex 提出技术方案。
4. 让 Codex 规划文件结构。
5. 确认方案后，再让 Codex 创建最小可运行版本。

注意事项：

- 不要一开始就要求“大而全”。
- 不要同时塞太多无关功能。
- 先做最小可运行版本，再逐步迭代。

## 二、让 Codex 修改代码时怎么控制范围

每次让 Codex 修改代码，都要先说明目标，再明确范围和限制。

可以这样问：

```text
请开始实现这个功能。

修改要求：
1. 可以修改哪些文件；
2. 不要修改哪些文件；
3. 不新增无关功能；
4. 不重构整个项目；
5. 不引入第三方库；
6. 不影响已有功能；
7. 修改完成后告诉我改了哪些文件、每个文件改了什么、我应该如何测试。
```

常用控制点：

- 明确允许修改的文件。
- 明确不允许做什么。
- 明确不要重构整个项目。
- 明确不要引入新依赖。
- 明确不能破坏已有数据结构。
- 明确修改完成后必须说明改动和测试方法。

如果只想改文档，可以说：

```text
请只修改 README.md，不要修改其他代码文件。
```

如果只想做分析，可以说：

```text
请只分析，不要修改任何文件。
```

## 三、功能包迭代方法

功能迭代不要太碎，也不要太乱。比较合适的方式是“一次做一组相关功能”。

适合放在一个功能包里的例子：

- 筛选、搜索、排序、重置筛选、显示结果数量
- JSON 导出、JSON 导入、导入预览、覆盖导入、合并导入
- 主题切换、主题保存、Toast、空状态、移动端优化
- 数据统计、情绪分布、最近 7 天、连续记录、本周回顾

不推荐的做法：

- 每次只做一个过小按钮，导致迭代太碎。
- 把完全无关的功能混在一起，例如同时做登录、动画、导入导出和重构。
- 在功能还没验收前继续叠加新功能。

每个功能包都应该包含：

- 修改方案
- 允许修改的文件
- 不允许做的事
- 数据是否会变化
- 兼容旧数据的方法
- 完整测试清单

功能包提示词示例：

```text
我想做一个 V1.x 功能包，包括……

请先说明完整修改方案，不要直接修改代码。
请重点说明：
- 需要修改哪些文件；
- HTML 准备新增什么；
- CSS 准备新增什么；
- JavaScript 准备新增哪些函数；
- 如何保证不影响已有功能；
- 如何完整测试。
```

## 四、测试与自检方法

每次功能完成后，都要测试。测试分两类：自己手动测试，以及让 Codex 做验收。

本地手动测试：

- 打开页面确认没有报错。
- 测试新增功能。
- 测试旧功能是否仍然正常。
- 刷新页面确认数据恢复。
- 测试空数据、错误输入、取消操作。
- 测试移动端宽度。

让 Codex 做功能验收：

```text
请对刚刚新增的功能做一次完整验收。

要求：
1. 只分析，不要修改文件；
2. 检查以下功能是否正常：……
3. 检查是否影响已有功能：……
4. 如有问题，只列出问题和修复建议，不要直接修改。
```

如果发现问题，再让 Codex 小范围修复：

```text
请只修复刚才验收中发现的明确问题。

要求：
1. 不新增功能；
2. 不重构项目；
3. 不影响已有功能；
4. 修改完成后告诉我改了哪些文件，以及我应该如何验证。
```

注意事项：

- 不要让 Codex “顺便优化一下所有代码”。
- 不要在不清楚问题时直接让它大改。
- 验收和修复要分开。

## 五、GitHub 保存方法

项目做到重要阶段时，要用 GitHub 保存版本。

建议提交时机：

- 最小可运行版本完成后
- 每个功能包完成并验收后
- 视觉重设计完成后
- 代码整理完成后
- README、CHANGELOG、发布文档完成后
- GitHub Pages 发布前后

提交信息要清楚，例如：

```text
feat: add mood filter and search
feat: add json backup import export
style: improve healing journal layout
refactor: organize storage and render flow
docs: update readme and changelog
chore: add release checklist
```

GitHub Pages 发布后要线上测试：

- 页面能否打开。
- CSS 和 JS 是否加载。
- 图片路径是否正常。
- 本地存储功能是否正常。
- 导入导出是否正常。
- README 中的在线地址是否正确。

## 六、常用提示词模板

### 只分析，不修改

```text
请先分析当前项目，不要修改任何文件。

请重点检查：
1. ……
2. ……
3. ……

如有问题，只列出问题和修复建议。
```

### 请先说明方案

```text
我想增加……功能。

请先说明完整修改方案，不要直接修改代码。
请重点说明：
- 需要修改哪些文件；
- 准备新增哪些结构或函数；
- 如何兼容旧数据；
- 如何测试。
```

### 不要重构整个项目

```text
请严格控制修改范围。
不要重构整个项目。
不要引入第三方库。
不要影响已有功能。
```

### 只修改相关文件

```text
请只修改……文件。
不要修改其他文件。
修改完成后告诉我具体改了哪些部分。
```

### 修改完成后告诉我如何测试

```text
修改完成后请告诉我：
1. 修改了哪些文件；
2. 每个文件改了什么；
3. 新增或调整了哪些函数；
4. 我应该如何测试。
```

### 请做一次完整验收

```text
请对当前功能做一次完整验收。

要求：
1. 只分析，不要修改文件；
2. 检查新增功能是否正常；
3. 检查是否影响已有功能；
4. 检查是否有明显重复代码或小问题；
5. 如有问题，只列出问题和修复建议。
```

### 请只修复明确问题

```text
请只修复刚才验收中发现的明确问题。

要求：
1. 不新增其他功能；
2. 不重构项目；
3. 不影响已有功能；
4. 修改完成后告诉我改了哪些文件，以及我应该如何验证。
```

## 七、适合继续升级的方向

### 前端视觉升级

- 优化布局层级
- 增加响应式细节
- 改进空状态和加载状态
- 增加克制的动画反馈
- 做主题系统和暗色模式

### 数据导入导出

- txt 导出
- JSON 备份
- 导入前预览
- 合并导入和覆盖导入
- 导入错误提示

### 代码整理

- 集中常量配置
- 统一数据读写函数
- 统一刷新入口
- 删除重复逻辑
- 增强错误处理
- 补充基础可访问性

### React / TypeScript 进阶

- 把原生 JavaScript 项目改写为 React
- 用组件拆分页面
- 用 TypeScript 定义数据类型
- 练习状态管理
- 练习表单和列表渲染

### 后端数据库进阶

- 增加账号系统
- 使用数据库保存记录
- 做跨设备同步
- 设计 API
- 处理权限和数据安全

### 真实产品化进阶

- 用户调研
- 需求优先级排序
- 更完整的错误处理
- 更系统的可访问性测试
- 自动化测试
- 发布说明和用户文档

## 最重要的原则

- 先做最小可运行版本。
- 每次只做一组相关功能。
- 修改前先分析，确认后再执行。
- 明确允许改什么，不允许改什么。
- 完成后必须测试。
- 有问题只做小范围修复。
- 不要让项目失控地变复杂。
